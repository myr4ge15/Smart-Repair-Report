# ТЕХНИЧЕСКОЕ ЗАДАНИЕ
**На разработку автоматизированной системы генерации сервисных отчетов «Smart Repair Report»**

## 1. Введение

### 1.1. Цель создания системы
Разработка программного обеспечения для автоматического преобразования «сырых» заметок инженера-ремонтника (список замененных деталей, действий, идентификаторов) в стандартизированные, юридически и технически грамотные формулировки для актов выполненных работ, внутренних отчетов и коммуникации с клиентами.

### 1.2. Проблематика
*   **Разнобой в терминологии:** Разные инженеры называют одну и ту же деталь по-разному (например, «микруха питания», «DC-DC», «AP3012»).
*   **Потеря контекста:** Важно не просто перечислить детали, но и указать статус прошивки и тестирования.
*   **Специфические правила бизнеса:** Некоторые действия (установка варистора) трактуются не как замена детали, а как «доработка узла».
*   **Ручной труд:** Сборка финальной фразы занимает время и подвержена ошибкам (забыли указать IMEI, ошиблись в падежах).

### 1.3. Пользователи
1.  **Инженер-ремонтник:** Вводит данные после ремонта.
2.  **Менеджер/Начальник сервиса:** Получает готовые тексты для актов и статистики.
3.  **Администратор базы знаний:** Обновляет справочники компонентов и правил.

## 2. Функциональные требования

### 2.1. Модуль ввода данных (Input Processor)
Система должна принимать входные данные в свободной форме.
**Поддерживаемые форматы ввода:**
1.  **Текстовая строка:** Копипаст из чата или черновика.
    *   *Пример:* `замена Модуль SIM7600E-H imei:868140072235570 , Микросхема AP3012, обновлена прошивка`
2.  **Структурированный список (чек-лист):** Выбор категорий и ввод партномеров через UI.
3.  **Пакетный импорт:** Загрузка CSV/Excel файла со столбцами: `DeviceID`, `ComponentsList`, `Actions`.

### 2.2. Ядро анализа и нормализации (Core Engine)
Это «мозг» приложения. Оно должно выполнять последовательный разбор входных данных.

#### 2.2.1. Словарь-маппер (Component Mapper)
База данных, связывающая **Паттерны поиска** (Part Number, название) с **Функциональной группой** и **Стандартной фразой**.

**Структура записи в БД компонентов:**
```json
{
  "id": "comp_001",
  "patterns": ["STM32L151", "GD32F103", "STM32F4"], // Регулярные выражения или подстроки
  "category": "MCU", // Категория для сортировки
  "standard_name": "микроконтроллер", // Существительное в именительном падеже
  "action_phrase": "Замена микроконтроллера", // Готовая фраза действия
  "priority": 1 // Порядок вывода в отчете (1 - самое важное)
}
```

#### 2.2.2. Распознавание специальных сущностей (Entity Extraction)
Помимо компонентов, система должна выявлять:
1.  **IMEI:**
    *   Паттерн: `imei[:\s]*(\d{15})`
    *   Действие: Сохранить значение в переменную `current_imei`. Привязать к категории «Модем».
2.  **Количество единиц:**
    *   Паттерн: `(\d+)\s*(шт|pcs)` или `- 2шт.`
    *   Действие: Если количество > 1, модифицировать фразу: «Замена транзисторов (2 шт.)» вместо «Замена транзистора».
3.  **Статусные маркеры:**
    *   `обновлена прошивка` / `прошивка` -> Флаг `firmware_updated = true`.
    *   `протестирован` / `тест` -> Флаг `tested = true`.
    *   `откат невозможен` -> Специальный флаг для добавления примечания.

#### 2.2.3. Логика «Доработок» и исключений
Некоторые компоненты не являются «заменами» в классическом смысле.
*   **Правило Варистора:** Если обнаружен паттерн варистора (напр., `B722...`), то действие не «Замена варистора», а **«Доработка встроенного блока питания»**.
*   **Правило Платы БП:** Замена всей платы БП трактуется как **«Замена встроенного блока питания»**.

### 2.3. Алгоритм сборки отчета (Report Builder)

После парсинга система формирует итоговый текст по строгому алгоритму:

**Шаг 1. Группировка и сортировка**
Все найденные действия сортируются по приоритету категорий:
1.  **Связь (Modem):** Если есть, выделяется отдельно из-за IMEI.
2.  **Основные узлы (MCU, Main PCB).**
3.  **Питание (Power IC, DC-DC, LDO, Varistor mods).**
4.  **Интерфейсы (RS485, USB, Ethernet).**
5.  **Периферия/Мелочевка (Transistors, Resistors, Connectors).**
6.  **Элементы питания (Batteries).**

**Шаг 2. Генерация основной части**
*   Берутся уникальные действия из отсортированного списка.
*   Объединяются через запятую.
*   *Пример логики:* `[Действие 1], [Действие 2], [Действие 3]`

**Шаг 3. Добавление стандартных хвостов**
*   Если `firmware_updated == true`: добавить `, обновлена прошивка`.
*   Если `tested == true`: добавить `, устройство протестировано`.

**Шаг 4. Обработка IMEI**
*   Если в группе «Связь» был найден IMEI:
    *   Основной текст заканчивается точкой.
    *   Добавляется перенос строки.
    *   Добавляется блок: `"Новый IMEI: [значение]"`

**Шаг 5. Формирование варианта «Требуется замена»**
*   Аналогично Шагу 2, но используется шаблон `требуется замена [имя компонента в Р.п.]`.
*   Исключаются фразы про прошивку и тест (так как это план работ).

### 2.4. Интерфейс пользователя (UI)

#### Экран «Генератор»
1.  **Поле ввода (Textarea):** Большое поле для вставки сырого текста. Поддержка автодополнения (autocomplete) при вводе партномеров.
2.  **Кнопка «Анализировать»:** Запускает парсинг.
3.  **Блок предпросмотра (Preview):**
    *   Показывает распознанные компоненты (с возможностью ручной корректировки, если робот ошибся).
    *   Показывает извлеченный IMEI.
4.  **Результат (Output):**
    *   Вкладка **«Отчет о работе»** (готовый текст для акта).
    *   Вкладка **«Дефектовка»** (текст «требуется замена...»).
    *   Кнопка **«Копировать»** (копирует в буфер обмена).
    *   Кнопка **«Очистить»**.

#### Экран «Администрирование базы знаний»
1.  Таблица компонентов: Поиск, Добавление, Редактирование.
2.  Поля карточки компонента:
    *   `Part Number / Название`
    *   `Категория` (выпадающий список: MCU, Power, Modem, etc.)
    *   `Фраза для отчета` (по умолчанию генерируется автоматически, но можно переопределить).
    *   `Приоритет сортировки` (число).
    *   `Флаг "Доработка"` (если да, меняет логику формирования фразы).

## 3. Структура данных (Обновленная и полная)

Для реализации требуется следующая схема базы данных (или JSON-структуры):

### 3.1. Таблица `components_dict` (Справочник компонентов)
| Поле | Тип | Описание | Пример |
| :--- | :--- | :--- | :--- |
| `id` | INT | Уникальный ID | 101 |
| `pattern` | STRING | Regex или substring для поиска | `AP3012KTR` |
| `category_code` | STRING | Код категории для сортировки | `POWER_DCDC` |
| `display_name` | STRING | Название для человека | `Микросхема AP3012` |
| `report_phrase` | STRING | Готовая фраза действия | `замена микросхемы DC-DC преобразователя` |
| `request_phrase` | STRING | Фраза для дефектовки | `микросхемы DC-DC преобразователя` |
| `is_modification` | BOOL | Это доработка, а не замена? | `FALSE` |
| `priority` | INT | Вес сортировки (меньше = выше) | 30 |

### 3.2. Таблица `categories` (Справочник категорий)
| Код | Название для сортировки | Стандартный суффикс действия |
| :--- | :--- | :--- |
| `MODEM` | Беспроводной модуль | `Замена беспроводного модуля связи` |
| `MCU` | Микроконтроллер | `Замена микроконтроллера` |
| `POWER_MAIN` | Блок питания (плата) | `Замена встроенного блока питания` |
| `POWER_IC` | Микросхемы питания | `Замена микросхемы управления питанием` |
| `INTERFACE` | Интерфейсы | `Замена микросхемы интерфейса` |
| `GPIO` | Транзисторы/Логика | `Замена транзисторов GPIO` |
| `BATTERY` | Элементы питания | `Замена элемента питания` |
| `MODIF` | Доработки | `Доработка встроенного блока питания` |

### 3.3. Объект сессии обработки (В памяти)
```json
{
  "input_text": "...",
  "detected_items": [
    {
      "component_id": 101,
      "phrase": "замена микросхемы DC-DC преобразователя",
      "category": "POWER_IC",
      "quantity": 1
    },
    {
      "component_id": 205,
      "phrase": "Замена беспроводного модуля связи",
      "category": "MODEM",
      "imei": "868140072235570"
    }
  ],
  "flags": {
    "firmware_updated": true,
    "tested": true
  }
}
```

## 4. Сценарии использования (Use Cases)

### Сценарий 1: Стандартная замена модема
**Вход:** `замена Модуль SIM7600E-H imei:868140072235570 , обновлена прошивка, протестирован`
**Логика:**
1.  Найдено `SIM7600E-H` -> Категория MODEM.
2.  Найдено `imei:...` -> Сохранено.
3.  Найдено `обновлена прошивка` -> Флаг True.
4.  Найдено `протестирован` -> Флаг True.
**Выход:**
```text
"Замена беспроводного модуля связи, обновлена прошивка, устройство протестировано.

Новый IMEI: 868140072235570"
```

### Сценарий 2: Сложный ремонт с доработкой
**Вход:** `замена Микроконтроллер STM32L151VBT6A, Варистор B72207S0271K101, обновлена прошивка`
**Логика:**
1.  `STM32...` -> MCU (Приоритет 1). Фраза: "Замена микроконтроллера".
2.  `B722...` -> VARISTOR. Флаг `is_modification=True`. Категория MODIF. Фраза: "Доработка встроенного блока питания".
3.  Сортировка: MCU (1) -> MODIF (5).
4.  Флаг прошивки: True.
**Выход:**
```text
Замена микроконтроллера, доработка встроенного блока питания, обновлена прошивка, устройство протестировано.
```
*(Примечание: "устройство протестировано" добавляется по умолчанию, если флаг теста не указан явно как ложный, либо требует явного указания во входных данных. В данном ТЗ предполагаем, что если написано "протестирован" - ставим флаг, если нет - не ставим. Или всегда ставим, если это финал ремонта. Лучше сделать флаг явным).*

### Сценарий 3: Группировка одинаковых компонентов
**Вход:** `Транзистор 2N7002 - 2шт., Транзистор IRLML0060 - 1шт.`
**Логика:**
1.  Оба относятся к категории GPIO.
2.  Система может либо перечислить их отдельно, либо объединить, если они считаются однотипной работой.
3.  *Рекомендация:* Если категория одна, но партномера разные — перечисляем через запятую внутри группы или суммируем количество, если фраза общая.
    *   Вариант А (детальный): `замена транзисторов 2N7002 (2 шт.) и IRLML0060`
    *   Вариант Б (общий, предпочтительный для актов): `замена транзисторов GPIO (3 шт.)`
    *   *Настройка:* В админке выбрать режим «Детальный» или «Обобщенный».

## 5. Технические требования к реализации

### 5.1. Стек технологий (Рекомендация)
*   **Frontend:** React.js или Vue.3 (для реактивного интерфейса).
*   **State Management:** Pinia/Vuex или Redux (для хранения состояния текущего отчета).
*   **Backend (Optional):** Python (FastAPI). Можно реализовать полностью на Client-Side (JS/WASM) для скорости и безопасности данных (данные не уходят на сервер).
*   **Database:** LocalStorage/IndexedDB (для браузера) или SQLite (для десктопа).

### 5.2. Требования к коду
*   Модульность: Парсер, Маппер, Генератор должны быть разнесены по разным модулям.
*   Покрытие тестами: Юнит-тесты на всех примерах из истории чата.

### 5.3. Безопасность
*   Если приложение веб-версии: работа без отправки данных на внешний сервер (все вычисления в браузере), так как IMEI и детали ремонтов могут быть конфиденциальны.

## 6. План разработки (Roadmap)

1.  **Этап 1: Прототип ядра (MVP)**
    *   Создание JSON-базы с основными компонентами из истории чата.
    *   Реализация простого парсера на JS.
    *   Верстка поля ввода и вывода.
2.  **Этап 2: Логика сборки**
    *   Реализация сортировки по приоритетам.
    *   Реализация обработки IMEI.
    *   Реализация правил «Доработок» (варисторы).
3.  **Этап 3: Интерфейс администратора**
    *   Возможность добавлять новые компоненты без правки кода.
    *   Редактирование шаблонов фраз.
4.  **Этап 4: Тестирование и полировка**
    *   Проверка на сложных кейсах (множество компонентов, разные количества).
    *   Добавление кнопки «Копировать» и истории действий.

---

## 7. Критерии приемки (Acceptance Criteria)

1.  Приложение корректно обрабатывает **100%** примеров, предоставленных в исходном чате.
2.  При замене модема IMEI всегда отображается в отдельном блоке снизу.
3.  Установка варистора всегда заменяет фразу «замена варистора» на «доработка встроенного блока питания».
4.  Пользователь может добавить новый компонент (например, новый тип датчика) через интерфейс админа, и он сразу начнет распознаваться в отчете.
5.  Результат готов к копированию в Excel/Word без необходимости ручного редактирования регистров и пунктуации.
